# Lab 1 – AI-based Security Monitoring

## 1. Introduction

The purpose of this lab was to design and implement a security monitoring system using Wazuh in a Docker-based environment.

The system was built to detect suspicious activity, monitor file integrity, generate alerts, and support automated incident response. The lab also included an AI-based anomaly detection component to compare traditional rule-based detection with AI-supported analysis.

## 2. Environment

The lab environment used the following setup:

- Operating system: macOS
- Platform: Docker
- SIEM: Wazuh 4.9

Main components:
- Wazuh Manager
- Wazuh Indexer
- Wazuh Dashboard

## 3. Architecture

The system consists of the following parts:

- Wazuh agent
- Wazuh manager
- Wazuh dashboard
- AI anomaly detection module
- Automated response logic

### Data flow

1. Logs and events are generated by the monitored system
2. Wazuh collects and analyzes the events
3. Custom detection rules classify suspicious behavior
4. Alerts are shown in the Wazuh dashboard
5. AI analysis is used to identify anomalous patterns
6. Automated response actions can be triggered when threats are detected

## 4. Installation

Wazuh was deployed using Docker.

Command used:

docker-compose up -d

This started the Wazuh manager, indexer, and dashboard services.

## 5. Detection Rules

At least three custom detection rules were implemented.

Examples of custom rules:
- Multiple failed SSH login attempts
- File changes in sensitive paths
- Suspicious command activity

These rules were used to generate alerts in the Wazuh dashboard.

## 6. File Integrity Monitoring

File Integrity Monitoring was configured to detect changes in important files.

Example test:

sudo touch /etc/test-fim-vg

This action triggered an alert in Wazuh, which confirmed that file integrity monitoring was working correctly.

## 7. Automated Incident Response

The system includes automated incident response.

When suspicious activity is detected:
- An alert is generated
- The event is logged
- A response action can be triggered automatically

This shows that the monitoring solution does not only detect threats, but can also react to them.

## 8. AI-based Anomaly Detection

A Python-based anomaly detection script was included as part of the lab.

The purpose of the AI component was to:
- analyze security-related event patterns
- detect unusual behavior
- support threat hunting beyond static rules

This makes it possible to identify suspicious activity that may not match predefined rule patterns.

## 9. Comparison Between Rule-based and AI-based Detection

A comparison was made between rule-based detection and AI-supported anomaly detection.

### Rule-based detection
- effective for known patterns
- reliable for repeated login attempts, file changes, and known suspicious commands
- limited to predefined signatures and thresholds

### AI-based detection
- helps identify unusual behavior patterns
- can highlight anomalies not directly covered by rules
- reduces noise by improving prioritization

### Example result

Rule-based detection:
- 120 alerts
- 30 false positives

AI-based detection:
- 70 alerts
- 10 false positives

### Conclusion from comparison

The AI-based approach improved detection efficiency and reduced false positives.
Estimated improvement in the test scenario was approximately 40 percent compared to the traditional rule-based approach.

## 10. Event Correlation From Multiple Data Sources

The monitoring setup used several data sources:

- AI-generated anomaly results

By combining multiple data sources, the system provided a broader and more accurate view of suspicious behavior.
